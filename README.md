# Gambit Dashboard

Инструмент **мастеров** Военно-Политической Игры (play-by-post в Discord): управление проектами,
сущностями, вердами (посты-сводки), регистрацией игроков и уведомлениями.

## Состав

| Компонент | Стек | Каталог |
|---|---|---|
| Backend API | FastAPI + SQLAlchemy 2 (async) + Postgres + Alembic | `backend/` |
| Discord-бот | discord.py (вебхуки + слэш-команды) | `bot/` |
| Frontend | React + Vite + TS (nginx в проде) | `frontend/` |

## Возможности

- **Проекты** и привязка Discord-каналов. `guild_id` проекта = Discord-сервер (по нему бот
  определяет проект для команд).
- **Сущности** с типами и **Jinja2-шаблонами embed** (кириллица), иерархия, закрепление за игроком
  (Discord user id), картинка сущности.
- **Описания страницами**: у типа несколько страниц, в Discord игрок листает их кнопками — так
  статы длиннее лимита эмбеда всё-таки помещаются. У каждой страницы свой **цвет полосы эмбеда**.
  Отдельной сущности можно включить **особое описание**, которое замещает шаблон её типа.
  В редакторе описания подсвечен синтаксис Jinja, а правая кнопка мыши вставляет атрибут,
  формулу или особую переменную из тех, что у сущности есть.
- **Особые переменные описания** помимо `label`: `{{ тип }}`, `{{ лидер }}`,
  `{{ игроки | строки("{имя} — {роль}") }}`, `{{ родители }}`, `{{ дети }}` и связи по типу —
  `{{ связи.союзник | строки("{название}") }}`. Настоящий атрибут с тем же именем всегда важнее;
  полный доступ — через `{{ сущность.игроки }}`.
- **Поля связанных сущностей** прямо в описании: элемент связи — это вся вторая сторона, с её
  атрибутами и формулами. Страна перечисляет свои структуры одной строкой —
  `{{ связи.принадлежит | строки("{название} — {описание}") }}`, доступны также
  `{тип_сущности}`, `{атрибуты.путь.через.точку}` и `{выч.мощь}`.
- **Связи сущностей** с произвольным типом, двух родов: **взаимные** (`союзник`, `война`) —
  стороны равны и видят связь одинаково — и **иерархические** (`состав`, `вассал`, галочка
  «родитель → дочерняя»), которые дают `{{ родители }}` и `{{ дети }}` и проверяются на циклы.
  По типу в шаблоне доступны обе. Отдельная вкладка проекта с графом (иерархия — стрелкой,
  взаимная связь — линией; узлы перетаскиваются, клик подсвечивает, фильтр по типу) и списком
  для правки типа и рода связи.
- **Вычисляемые поля**: формулы от атрибутов, заданные путём с точками
  (`бюджет.деньги = казна.прирост - казна.расход`). Складываются в дерево и доступны в шаблоне
  как `{{ выч.бюджет.деньги }}`, вся ветка сразу — `{{ выч.бюджет | поля }}`. Считаются в дашборде
  на лету, ошибка в формуле не роняет карточку. Доступны и в правках вердов.
  В редакторе формулы складываются в такое же дерево по пути, как атрибуты (`бюджет` →
  `ресурсы`), с тем же переключателем глубины.
  Формулы задаются **у типа** и **у самой сущности**: собственные не замещают типовые, а
  дополняют их — совпадение путей переопределяет одну формулу, остальные продолжают работать
  (страна-исключение получает свой расчёт бюджета, не теряя остальных).
- **Атрибуты-списки** (национальные духи, союзники, гигаструктуры): правятся по строке на
  элемент, в шаблоне печатаются через запятую сами, а фильтры дают другие виды —
  `{{ духи | список }}`, `| нумерованный`, `| через_запятую(пусто="нет")`, `| сколько`,
  `{{ флот | строки("{имя} — {тоннаж}") }}`. Доступен и весь Jinja2 (`{% for %}`, `join`,
  `sort`, `map`). В формулах — `длина(духи)` и `сумма(флот, "тоннаж")`, в правках вердов —
  режимы «добавить»/«убрать» по одному элементу.
- **Редактор атрибутов**: строки складываются в дерево по пути на всю глубину
  (`РЕС.пища.запас` и `РЕС.пища.расход` — «РЕС» → «пища»; вложенность можно ограничить
  переключателем, а группа из одного поля не заводится), есть поиск,
  правка списков по строке на элемент и режим JSON для переноса структуры целиком.
- **Верды**: черновик → публикация/расписание. Публикация применяет правки атрибутов сущностей;
  до неё дашборд показывает по каждой правке «было → станет», посчитанное на копии данных.
  Бот отправляет верд **через вебхук** в любой канал (создаёт вебхук при необходимости), с подменой
  имени/аватара автора и опциональным эмбедом. Верд можно сохранить **шаблоном**, отметив, какие
  именно поля он переносит.
- **Речь от лица сущности**: команда `/say` (модалка с текстом) и **авто-подмена** — галочка на
  канале, после которой сообщения игрока переотправляются вебхуком под именем и аватаркой его
  сущности. Если в канал игрока пускают сразу несколько его сущностей, он выбирает одну командой
  `/say-as`, и выбор запоминается для этого канала.
- **Слэш-команды бота**: `/about` (карточка проекта), `/me-info` (карточка сущности),
  `/say`, `/say-as`, `/ping-master` (уведомление мастеру), `/register` (форма регистрации).
- **Регистрационные формы**: мастер конструирует поля (варианты ответа добавляются по одному,
  порядок полей меняется стрелками); заявки по каждой форме видны на её же экране. Решение —
  одобрить или отклонить **с текстом**, который бот отправляет игроку в личные сообщения
  (название проекта, формы и причина). Сущность при одобрении **не создаётся**: анкета редко
  совпадает с атрибутами, поэтому сущность заводится отдельно, а игрока к готовой можно
  привязать прямо в окне решения.
- **Уведомления**: пинги игроков и новые заявки прилетают во вкладку «Уведомления».

## Локальный запуск (Docker)

```bash
cp .env.example .env      # заполните пароли, DISCORD_BOT_TOKEN, INTERNAL_API_KEY
docker compose up --build
```

- Дашборд: http://localhost:8080 (nginx проксирует `/api` на backend)
- API/Swagger: http://localhost:8000/docs

Миграции применяются автоматически при старте backend-контейнера.

### Что включить в Discord Developer Portal

- **SERVER MEMBERS INTENT** — список участников сервера (выбор игроков в дашборде).
- **MESSAGE CONTENT INTENT** — авто-подмена сообщений. Без него бот **не запустится**, пока
  не поставить `BOT_MESSAGE_CONTENT=0` (тогда работает только `/say`).
- Право **Manage Webhooks** в каналах, где бот отправляет верды и подменяет сообщения.
- `PUBLIC_BASE_URL` (напр. `https://ваш-домен/api`) — **обязателен для аватарок**: Discord
  скачивает картинки сам, и без него загруженный файл останется внутренним путём, а сообщения
  от лица сущности уйдут с аватаркой по умолчанию. Дашборд предупреждает об этом на картинке
  сущности и на экране каналов (проверка — `GET /system/info`). Внешние ссылки
  (`https://…/avatar.png`) работают всегда.

## Разработка (без Docker)

```bash
# Backend (нужен запущенный Postgres, DATABASE_URL в окружении)
cd backend && pip install -r requirements.txt
alembic upgrade head
uvicorn app.main:app --reload

# Frontend
cd frontend && npm install && npm run dev   # http://localhost:5173, /api → :8000
```

## Развёртывание в Portainer (готовые образы из GHCR)

1. Запушьте репозиторий в GitHub. Workflow `.github/workflows/docker.yml` соберёт и опубликует
   образы `ghcr.io/<owner>/gambit-backend|bot|frontend` при пуше в `main` (или теге `v*`).
2. Убедитесь, что пакеты GHCR доступны (при необходимости сделайте их публичными или добавьте
   registry-креды в Portainer).
3. В Portainer → **Stacks → Add stack** вставьте `docker-compose.portainer.yml`, задайте переменные
   окружения из `.env.example` (обязательно `IMAGE_OWNER`, пароли, `INTERNAL_API_KEY`,
   `DISCORD_BOT_TOKEN`).
4. Deploy. Дашборд — на `FRONTEND_PORT` (по умолчанию 8080).

## Discord-бот: настройка

1. В [Developer Portal](https://discord.com/developers/applications) создайте приложение и бота,
   скопируйте **токен** в `DISCORD_BOT_TOKEN`.
2. Права бота: `Manage Webhooks`, `Send Messages`, а также `applications.commands` (слэш-команды).
3. Пригласите бота на сервер, укажите `guild_id` этого сервера в настройках проекта.
4. `INTERNAL_API_KEY` бота и backend **должны совпадать**.

## Безопасность

- Вход мастеров — по общему паролю (`MASTER_PASSWORD`), токен подписывается `SECRET_KEY`.
- Ручки `/internal/*` (для бота) защищены `INTERNAL_API_KEY`.
- Перед продом смените все значения из `.env.example` на случайные секреты.
